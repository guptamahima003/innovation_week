                    2. DATA INPUTS & SOURCES
================================================================================

  The system ingests three categories of customer data:

  A. 1ST-PARTY TRANSACTIONAL DATA
     - lifetime_order_count (1-100+)
     - lifetime_spend ($200-$35K)
     - avg_order_value ($30-$2000)
     - return_rate (0.0-0.35)
     - preferred_payment (credit_card, debit, paypal, financing)
     - loyalty_tier (none, member, silver, gold, elite)

  B. BEHAVIORAL EVENT STREAM (Real-Time via WebSocket)
     - browse: product page view
     - add_to_cart: item added to cart
     - remove_from_cart: item removed
     - search: search query executed
     - checkout_start: checkout flow initiated
     - checkout_complete: purchase completed
     - page_leave: navigation away from page
     - exit_intent: mouse leaving viewport

  C. CONTEXTUAL & DEMOGRAPHIC
     - estimated_household_income ($25K-$200K+)
     - home_ownership (rent, own, other)
     - household_size (1-6+)
     - has_children (boolean)
     - life_stage (student, young_professional, established_family, etc.)

                    3. ML PERSONA CLUSTERING PIPELINE
================================================================================

  STAGE 1: Training (happens once at server startup)
  -------------------------------------------------------
  - 10,000 synthetic customer profiles are loaded, each with 14 numerical
    behavioral features.
  - A K-Means clustering algorithm (k=10) is trained on these profiles.
    - Features are first normalized via StandardScaler (zero mean, unit variance)
      so that no single feature dominates due to scale differences.
    - K-Means finds 10 natural clusters in the 14-dimensional feature space.
  - Each of the 10 resulting cluster centroids is then mapped to a named
    persona using hand-crafted scoring rules. For example:
      - The cluster whose centroid has the highest
        (price_sensitivity + deal_seeking_score + cart_abandonment_rate)
        becomes "value_hunter".
      - The cluster whose centroid has the highest
        (tech_affinity - price_sensitivity) becomes "tech_enthusiast".
    - Rules are evaluated greedily in order; once a cluster is claimed by a
      persona, it cannot be claimed again.
  - The trained model (scaler + K-Means + cluster-to-persona mapping) is
    stored in memory as the PersonaEngine.
  - Silhouette score is computed and logged for quality assessment.

  STAGE 2: New User Prediction (per API call)
  -------------------------------------------------------
  When POST /api/customer is called with a new user's profile data:

    1. The 14 FEATURE_KEYS are extracted from the request:
       [tech_affinity, price_sensitivity, deal_seeking_score, research_depth,
        brand_loyalty_score, avg_order_value, lifetime_spend,
        total_site_visits_90d, avg_session_duration_min, cart_abandonment_rate,
        propensity_to_churn, days_since_last_visit, lifetime_order_count,
        return_rate]

    2. The feature vector is scaled using the SAME StandardScaler that was
       fit during training.

    3. K-Means finds which of the 10 clusters this feature vector is
       closest to (minimum Euclidean distance to centroid).

    4. The cluster index is looked up in cluster_to_persona to get the
       persona name (e.g., cluster 3 -> "value_hunter").

    5. A confidence score is computed using a softmax-style formula:
       confidence = (1/distance_to_assigned_cluster) /
                    sum(1/distance_to_each_cluster)

    6. The profile is stored in memory with the assigned persona_type and
       persona_confidence.

  STAGE 3: Lightweight Prediction (no persistence)
  -------------------------------------------------------
  POST /api/customer/predict-persona accepts just the 14 features and
  returns the predicted persona + confidence without creating a profile.

  THE 14 CLUSTERING FEATURES:
  -------------------------------------------------------
  FEATURE                    RANGE       DESCRIPTION
  -------------------------  ----------  ------------------------------------------
  tech_affinity              0.0 - 1.0   Affinity for technology products
  price_sensitivity          0.0 - 1.0   How sensitive to price changes
  deal_seeking_score         0.0 - 1.0   Active deal/coupon seeking behavior
  research_depth             0.0 - 1.0   How deeply they research before buying
  brand_loyalty_score        0.0 - 1.0   Tendency to stick with preferred brands
  avg_order_value            $30 - $2000  Average dollar value per order
  lifetime_spend             $200 - $35K  Total historical spend
  total_site_visits_90d      0 - 80       Site visits in last 90 days
  avg_session_duration_min   1 - 35 min   Average time per visit
  cart_abandonment_rate      0.0 - 1.0   Fraction of carts not completed
  propensity_to_churn        0.0 - 1.0   Likelihood of becoming inactive
  days_since_last_visit      0 - 180      Recency of last visit
  lifetime_order_count       1 - 100+     Total number of orders
  return_rate                0.0 - 0.35   Fraction of orders returned

  | Loyalty Power User       | MED  | LOW+ | LOW+ | MED  | HIGH | 320  | 12000 |
  | Lapsing Customer         | MED- | MED  | MED  | LOW+ | LOW+ | 250  | 3500  |
  | Business Buyer           | MED+ | LOW  | LOW  | MED+ | MED  | 800  | 15000 |

  4.1 TECH ENTHUSIAST (Color: #6366f1)
  Description: Early adopter with high tech affinity and above-average order
               values. Seeks the latest gadgets and innovations.
  Clustering rule: Maximizes (tech_affinity - price_sensitivity)

  4.2 VALUE HUNTER (Color: #f59e0b)
  Description: Price-sensitive shopper who actively seeks deals, compares
               prices, and has high cart abandonment at full price.
  Clustering rule: Maximizes (price_sensitivity + deal_seeking + cart_abandon)

  4.3 CONSIDERED RESEARCHER (Color: #10b981)
  Description: Deep researcher who reads reviews, compares specs, and takes
               time before committing. Low impulse purchases.
  Clustering rule: Maximizes (research_depth + session_duration/30)

  4.4 LOYALTY POWER USER (Color: #3b82f6)
  Description: High-LTV customer with strong brand loyalty and active
               rewards membership. Multi-category buyer.
  Clustering rule: Maximizes (brand_loyalty + lifetime_spend/20K - churn)

  4.5 LAPSING CUSTOMER (Color: #ef4444)
  Description: Previously active customer showing declining engagement.
               High churn risk requiring win-back strategies.
  Clustering rule: Maximizes (churn + days_since_visit/100 - visits/50)

  4.6 BUSINESS BUYER (Color: #8b5cf6)
  Description: B2B purchaser buying in bulk with high AOV. Values service
               SLAs and managed solutions over consumer deals.
  Clustering rule: Maximizes (AOV/1K + lifetime_spend/20K - deal_seeking)

  4.7 IMPULSE BUYER (Color: #ec4899)
  Description: Fast decision-maker with low research depth and high
               purchase rate. Responds strongly to urgency and scarcity.
  Clustering rule: Maximizes (-research_depth - session_duration/30 + cart_abandon)

  4.8 HOME UPGRADER (Color: #14b8a6)
  Description: Focused on appliances and smart home products. Mid-to-high
               AOV, family-oriented, seasonal buyer.
  Clustering rule: Maximizes (AOV/1K + 0.5*tech_affinity - deal_seeking)

  4.9 GIFT SHOPPER (Color: #f97316)
  Description: Seasonal buyer who browses outside usual categories. Checks
               reviews for unfamiliar products and values gift-ready options.
  Clustering rule: Maximizes (research*0.5 + price_sens*0.3 + brand_loyalty*0.2)

  4.10 STUDENT BUDGET (Color: #06b6d4)
  Description: Price-sensitive younger demographic focused on education and
               entertainment categories. Financing-inclined with limited spend.
  Clustering rule: Maximizes (price_sensitivity - lifetime_spend/20K - AOV/1K)

                    5. PERSONA TRANSITION & DYNAMIC RE-BUCKETING
================================================================================

  Personas are NOT static. The system re-evaluates a customer's persona when
  their behavior changes significantly:

  HOW IT WORKS:
  1. Every event (browse, add_to_cart, search, checkout, page_leave, etc.)
     increments a per-customer event counter.

  2. When the counter reaches the REASSIGNMENT_EVENT_THRESHOLD (default: 20
     events), the reassignment pipeline fires:

     a. _update_profile_from_history() aggregates the customer's real session
        data and updates their profile's behavioral fields:
        - total_site_visits_90d -- updated to reflect actual activity
        - search_queries_90d -- from observed search events
        - cart_additions_90d -- from observed add_to_cart events
        - days_since_last_visit -- set to 0 (currently active)
        - cart_abandonment_rate -- recalculated from actual cart vs checkout
          completions, blended 60/40 with historical rate
        - propensity_to_churn -- reduced by 30% if >10 events observed

     b. PersonaEngine.predict() is called with the updated profile.

     c. If the persona CHANGED:
        - profile's persona_type and persona_confidence are updated
        - Dashboard persona_distribution counters are adjusted
        - Change is broadcast to all connected dashboard clients

     d. Event counter resets to 0 and the cycle begins again.

  CONFIGURATION:
    REASSIGNMENT_EVENT_THRESHOLD: 20 events (configurable in config.py)

                    6. ABANDON DETECTION & CLASSIFICATION
================================================================================

  DETECTION RULES (AbandonDetector):
  -----------------------------------
  - page_leave from /cart with items -> CART_ABANDON
  - page_leave from /checkout without completion -> CHECKOUT_ABANDON
  - page_leave from /product with >10s dwell and no add_to_cart -> PRODUCT_PAGE_ABANDON
  - exit_intent with cart items -> CART_ABANDON or CHECKOUT_ABANDON
  - search with results but no click -> SEARCH_ABANDON

  CLASSIFICATION MATRIX (ReasonClassifier):
  -------------------------------------------
  (abandon_type x persona_type) -> abandon_reason

  +-------------------------+-------------+-------------+-------------+-------------+
  | PERSONA                 | Cart        | Checkout    | Product Pg  | Search      |
  |                         | Abandon     | Abandon     | Abandon     | Abandon     |
  +-------------------------+-------------+-------------+-------------+-------------+
  | Tech Enthusiast         | Confidence  | Payment     | Confidence  | Cant Find   |
  | Value Hunter            | Price       | Payment     | Price       | Cant Find   |
  | Considered Researcher   | Confidence  | Payment     | Confidence  | Cant Find   |
  | Loyalty Power User      | Distraction | Payment     | Distraction | Cant Find   |
  | Lapsing Customer        | Distraction | Price       | Distraction | Cant Find   |
  | Business Buyer          | Price       | Payment     | Confidence  | Cant Find   |
  | Impulse Buyer           | Distraction | Payment     | Distraction | Cant Find   |
  | Home Upgrader           | Price       | Payment     | Confidence  | Cant Find   |
  | Gift Shopper            | Confidence  | Payment     | Confidence  | Cant Find   |
  | Student Budget          | Price       | Payment     | Price       | Cant Find   |
  +-------------------------+-------------+-------------+-------------+-------------+

  CONTEXTUAL OVERRIDES (applied before the matrix lookup):
  - If price_sensitivity > 0.7 AND cart_total > 1.5x avg_order_value
    -> Override to PRICE_TOO_HIGH
  - If research_depth > 0.7 AND time_on_page > 20 seconds
    -> Override to CONFIDENCE_GAP
  - If deal_seeking_score > 0.7 AND cart abandon
    -> Override to PRICE_TOO_HIGH

                    8. PERSONALIZATION LAYER (STOREFRONT)
================================================================================

  The storefront applies persona-driven personalization across three surfaces:

  A. PRODUCT LISTING PAGE (PLP) - SPARSE APPROACH
  ------------------------------------------------
  NOT every product card gets personalized treatment. The system uses a
  seeded random function (deterministic per product ID) to decide:

  - Social proof badges: ~30% of cards get a badge (shouldShowBadge)
  - Price hint sub-text: ~40% of cards get price context (shouldShowPriceHint)
  - Badge variants: 2-4 different badge texts per persona, selected
    deterministically so the same product always shows the same badge

  Persona badge examples:
    tech_enthusiast: "Editor's Choice", "Top Rated in Tech", "Staff Pick"
    value_hunter:    "Best Value", "Price Drop", "Deal of the Day"
    impulse_buyer:   "Selling Fast", "Almost Gone", "Hot Right Now"
    student_budget:  "Student Deal", "Budget Friendly"

  B. PRODUCT DETAIL PAGE (PDP)
  ------------------------------------------------
  - Social proof badge on product image (always shown on PDP)
  - Dynamic price tag + sub-text based on persona
  - Persona-specific CTA button (label, color, micro-copy)
  - WhyBuy module with persona-relevant purchase arguments

  CTA examples:
    tech_enthusiast:      "Pre-Order Now" (indigo)
    value_hunter:         "Grab This Deal" (yellow)
    considered_researcher:"Add to Cart" (emerald)
    impulse_buyer:        "Buy Now - Arrives Tomorrow" (orange-pink gradient)
    student_budget:       "Buy - Pay Monthly" (cyan)

  C. SESSION NUDGES (Toast-style, shown once per session)
  ------------------------------------------------
  After the user has browsed several products, a subtle toast notification
  appears with a persona-appropriate message:

    tech_enthusiast:      "New arrivals just dropped"
    value_hunter:         "Flash deals ending soon - up to 30% off"
    loyalty_power_user:   "You have 2X points on all purchases today"
    lapsing_customer:     "Welcome back! Here's 5% off your next order"

  D. ABANDON INTERVENTION OVERLAYS
  ------------------------------------------------
  Slide-up card triggered when abandon is detected. Shows:
  - Headline, body text, CTA button (all persona-specific)
  - Auto-dismisses after configurable duration (10-25 seconds)
  - Delivered via WebSocket from backend intervention engine

                    10. DEMO SIMULATOR
================================================================================

  The DemoSimulator runs at server startup to pre-populate dashboard metrics
  so the dashboard isn't empty during demos.

  CONFIGURATION:
    DEMO_SIMULATOR_ENABLED: true (env var)
    DEMO_SIMULATOR_SESSIONS: 200 (env var)

  HOW IT WORKS:
  1. Distributes 200 sessions across 10 personas using weighted distribution:
     tech_enthusiast: 13%, value_hunter: 15%, considered_researcher: 10%,
     loyalty_power_user: 8%, lapsing_customer: 8%, business_buyer: 5%,
     impulse_buyer: 13%, home_upgrader: 10%, gift_shopper: 10%,
     student_budget: 8%

  2. Each session:
     a. Creates a real session from the profile store (with force_persona)
     b. Records session in metrics tracker
     c. Picks a random product from the catalog
     d. Decides if session ends in conversion (persona-specific rates):
        - loyalty_power_user: 45%, impulse_buyer: 35%, tech_enthusiast: 30%
        - business_buyer: 25%, gift_shopper: 22%, home_upgrader: 20%
        - considered_researcher: 18%, value_hunter: 12%, student_budget: 10%
        - lapsing_customer: 8%
     e. Decides if session ends in abandon (persona-specific rates):
        - value_hunter: 70%, student_budget: 65%, lapsing_customer: 60%
        - considered_researcher: 55%, home_upgrader: 50%, business_buyer: 45%
        - gift_shopper: 45%, impulse_buyer: 40%, tech_enthusiast: 35%
        - loyalty_power_user: 25%

  3. For abandon sessions:
     - Picks abandon type using persona-specific weights
     - Builds AbandonSignal with realistic product/cart data
     - Runs through REAL classifier and intervention engine
     - Records metrics (abandon + intervention + revenue)

  TYPICAL OUTPUT:
    ~200 sessions, ~95 abandons, ~$143K revenue at risk, ~$19.7K recovered

                    11. DEPLOYMENT ARCHITECTURE
================================================================================

  The project deploys as two separate services:

  BACKEND (FastAPI / Python 3.11):
    Platform: Render.com (Web Service)
    Root: /backend
    Build: pip install -r requirements.txt
    Start: uvicorn main:app --host 0.0.0.0 --port $PORT
    Key deps: fastapi, uvicorn, scikit-learn, numpy, pandas, faker

  FRONTEND (Next.js 14 / Node 20):
    Platform: Render.com or Vercel
    Root: /frontend
    Build: npm ci && npm run build
    Start: npm start
    Output: standalone (for Docker/serverless)
    Env vars: NEXT_PUBLIC_API_URL, NEXT_PUBLIC_WS_URL

  COMMUNICATION:
    REST API: HTTPS (CORS configured)
    WebSocket: WSS for real-time events (storefront + dashboard)

  render.yaml Blueprint deploys both services together with auto-linking.

                    13. THE BIG PICTURE (NON-TECHNICAL SUMMARY)
================================================================================

Imagine you walk into a Best Buy store. A smart sales associate notices:

"This person always waits for sales" -> They show you a coupon
"This person loves the latest gadgets" -> They show you what just launched
"This person reads every review before buying" -> They hand you a comparison guide

This app does exactly that -- but online, automatically, and in real-time.

The system:
  1. Generates 10,000 realistic customer profiles (synthetic data)
  2. Uses ML to group them into 10 persona archetypes (K-Means clustering)
  3. Personalizes the storefront experience per persona (badges, CTAs, prices)
  4. Tracks every action on the storefront in real-time (WebSocket events)
  5. Detects when someone is about to leave (abandon detection)
  6. Figures out WHY they're leaving (reason classification by persona)
  7. Shows them the right message at the right moment (50 unique interventions)
  8. Re-evaluates their persona as behavior changes (dynamic reassignment)
  9. Tracks all metrics per-persona on a live dashboard with 8 KPIs
  10. Pre-populates with 200 simulated sessions for demo readiness

Same product, same abandon action, but completely different interventions
based on who the customer is. That's the core value proposition.
